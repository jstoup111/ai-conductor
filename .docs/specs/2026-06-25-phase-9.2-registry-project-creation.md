# PRD: Phase 9.2 — Project Registry + Creation

**Date:** 2026-06-25
**Status:** Approved
**Program:** Phase 9 (Supervisor/Engineer). Third slice — the registry the engineer reads + the
project-creation it will call. Based on `main` (independent of #82 / 9.1).
**Predecessor design:** `.docs/decisions/supervisor-engineer-followup.md` §9.2, §7

## Problem / Background

The engineer (9.3) needs to know **which projects exist** to route ideas and feed per-project
daemons, and needs a way to **create** a new project (scaffold + register). Today nothing tracks
projects across the machine, and onboarding a project (`/bootstrap`) leaves no cross-project
trace. 9.2 builds the **registry + registration + project-creation** now; the engineer that *reads*
the registry to plan/route is deferred to 9.3 (same "plumbing now, consumer later" pattern as 9.1).

Locked program decisions: **separate git repo per project**; the engineer holds a **registry of repo
paths**; project creation is available **both** ways — a human runs `/bootstrap` (which registers)
*and* a standalone create command scaffolds + registers (the engineer calls it in 9.3).

## Goals & Non-Goals

**Goals**
- A cross-project **registry** (`~/.ai-conductor/registry.json`) of project records, beside the
  harness user config + engineer store.
- **Registration**: `/bootstrap` auto-registers; a `register` command registers an existing repo.
- **Creation**: a standalone `create` command scaffolds a new project (git init + bootstrap
  CLAUDE.md + remote + `.gitignore`) and registers it.

**Non-Goals**
- **No engineer / no reader-that-plans** — no "engineer reads registry at plan time", no routing, no
  daemon orchestration. A types-only reader interface is exported for 9.3; no consumer behavior.
- **No list/inspect CLI** (deferred — reader is 9.3).
- No daemon-state *transitions* (the field exists; 9.2 sets only initial values).
- No multi-host, no registry UI, no credential storage.

## Users / Personas

- **The future engineer (9.3)** — reads the registry to route + create; 9.2 produces what it consumes.
- **The human operator** — runs `/bootstrap` (auto-registers) or `create` to stand up a project;
  benefits from a single source of truth for "what projects exist."

## Functional Requirements

- **FR-1:** A registry file at **`~/.ai-conductor/registry.json`** (overridable via
  `$AI_CONDUCTOR_REGISTRY`), created if absent, holding a versioned list of project records.
  Never inside a project repo.
- **FR-2:** Each record: `{schemaVersion, name, path (absolute), remote?, status, daemonState?,
  lastSignalRef?, registeredAt}`. (Exact types locked at architecture/ADR.) `status` ∈
  {`registered`, `created`}; `daemonState`/`lastSignalRef` are present-but-initial (9.3 transitions
  them).
- **FR-3:** A **`register [path]`** command (default `path` = cwd) records the project: `name` =
  basename, absolute `path`, `remote` discovered from git if present, `status: registered`.
- **FR-4:** *(edge)* `register` is **idempotent** — re-registering an existing `path` **updates**
  that record in place (dedup by absolute path), never appends a duplicate.
- **FR-5:** *(edge)* `register` on a **non-existent or non-git** path fails with a clear error and
  leaves the registry **unchanged** (no partial/corrupt write).
- **FR-6:** A **`create <name> [--remote <url>]`** command scaffolds a new project: `git init`,
  a bootstrap `CLAUDE.md` (from the template, referencing HARNESS.md), seed `.gitignore` with
  `.pipeline/`, `.daemon/`, `.worktrees/`, set the remote if given — then registers it
  (`status: created`).
- **FR-7:** *(edge)* `create` **refuses to clobber**: if the target directory exists and is
  non-empty, it errors and writes nothing (no scaffold, no registry entry).
- **FR-8:** `/bootstrap` **auto-registers** the onboarded project (invokes the same registration
  path) so existing projects enter the registry without a separate manual step.
- **FR-9:** *(edge)* Registry writes are **atomic and concurrency-safe** (temp-file + rename;
  concurrent registrations don't corrupt the JSON). Unlike the engineer store, a registry write
  failure is a **reported error** (registration is a deliberate user action), not silently
  swallowed.
- **FR-10:** A **types-only** `RegistryReader` interface (`listProjects()`, `getProject(path)`) +
  the `ProjectRecord` type are exported for 9.3. No runtime consumer behavior ships in 9.2.
- **FR-11:** *(edge)* The registry stores **no secrets** — `remote` is the git remote URL as-is;
  no tokens/credentials are captured or written.

## Non-Functional Requirements

- **Atomic, concurrency-safe writes** (temp + rename) — multiple writers (bootstrap, register,
  create) and future daemons.
- **Reuse harness conventions** — registry lives under `~/.ai-conductor/` beside `config.yml` and
  `engineer/`; path resolution mirrors `user-config.ts` (injectable home/env, testable).
- **Forward-compatible** — `schemaVersion` per record/file so the 9.3 reader can evolve.
- **No partial state** — a failed register/create leaves the registry and filesystem unchanged.

## Acceptance Criteria / Success Metrics

- Every FR covered by passing tests (vitest):
  1. `register` in a git repo → record with name/abs-path/remote/status; re-register → updated, not
     duplicated.
  2. `register` on a non-git/missing path → error, registry unchanged.
  3. `create <name>` → scaffolds git repo + CLAUDE.md + `.gitignore` (with the three ignores) +
     remote, and a `created` record appears.
  4. `create` into a non-empty dir → refuses, nothing written.
  5. Registry path defaults to `~/.ai-conductor/registry.json`, honors `$AI_CONDUCTOR_REGISTRY`,
     auto-creates, and is outside any project repo.
  6. Concurrent registrations → registry stays valid JSON with all records (atomic writes).
  7. `RegistryReader`/`ProjectRecord` types exported (compile-time); no consumer behavior.
  8. `/bootstrap` registration step present (skill updated) and idempotent.
- Existing tests still pass; harness integrity green; CHANGELOG `[Unreleased]` +
  README/`src/conductor/README.md` updated (docs-track-features); bootstrap SKILL.md change passes
  validation.

## Scope

### In Scope
- The registry module (path resolution, schema, atomic read/write, dedup-by-path).
- `register` + `create` CLI commands (conduct-ts).
- `/bootstrap` auto-register step.
- Types-only reader interface for 9.3. Tests + docs.

### Out of Scope
- The engineer / registry reader behavior / routing / daemon orchestration (9.3).
- List/inspect CLI; daemon-state transitions; budgets enforcement; multi-host; UI.

## Key Decisions & Rationale

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Registry + registration + create now; **reader deferred to 9.3** | The engineer doesn't exist yet; ship the plumbing, keep it testable in isolation (mirrors 9.1). |
| 2 | **`~/.ai-conductor/registry.json`** (dedicated JSON) | Beside the user config + engineer store; JSON is safely machine-rewritable by multiple writers, unlike hand-edited `config.yml`. |
| 3 | **Both** create paths (`create` command + bootstrap auto-register) | Locked program decision; the standalone `create` is what the engineer calls in 9.3, and bootstrap captures human-onboarded projects. |
| 4 | Registry writes **report errors** (not best-effort) | Registration is a deliberate user action — a silent failure would lose a project; contrast the engineer store (best-effort, must never break a ship). |
| 5 | **Atomic temp+rename**, dedup by absolute path | Multiple writers; no partial/duplicate/corrupt state. |
| 6 | Commands live in **conduct-ts** (not bin/conduct bash) | conduct-ts is the active PRD-driven conductor per HARNESS.md. |

## Dependencies

- Harness user-config conventions (`~/.ai-conductor/`, `user-config.ts` path pattern).
- conduct-ts CLI (`src/conductor` `index.ts`/`cli.ts`), the `CLAUDE.md`/bootstrap templates,
  `skills/bootstrap`.
- `git` (init, remote discovery). No new external deps.
- **Independent of 9.1/#82.** The registry↔engineer-store link (a record's `lastSignalRef`; and 9.1's
  engineer key adopting the registry's canonical project id — 9.1 retro A-1) is **deferred** until
  both are merged (a small follow-up or part of 9.3).

## Open Questions

- None blocking. The **record schema field types**, the **atomic-write mechanism**, and the
  **bootstrap→register invocation** (CLI call vs shared lib) are architecture/ADR decisions.
- Whether `create`'s remote handling should `git remote add` only, or also attempt an initial push
  — lean **add-only** (no network side effects in `create`); confirm at architecture.
