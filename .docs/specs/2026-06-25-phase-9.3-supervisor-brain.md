# PRD: Phase 9.3 — Supervisor / Brain

**Date:** 2026-06-25
**Status:** Approved
**Program:** Phase 9 (Supervisor/Brain). Capstone — the consumer that ties 9.1 (brain store) +
9.2 (registry) together. Based on `main` (incl. 9.0/9.1/9.2).
**Predecessor design:** `.docs/decisions/supervisor-brain-followup.md` §9.3, §1, §2, §5

## Problem / Background

The plumbing now exists — a per-project daemon ships backlogs (9.0), emits structured retro
signals to a cross-project store (9.1), and a registry tracks projects (9.2). What's missing is
the **brain**: a centralized, **non-autonomous** planning service that takes a human's feature
idea, routes it to the right project, authors a spec there, and hands a human a reviewable
spec — and that **learns** from the accumulated retro signals so successive plans get better
(the flywheel). The brain is the **backlog producer** the daemon was built to receive; it
**never builds and never auto-merges**.

## Goals & Non-Goals

**Goals**
- An interactive **`conduct brain`** loop: idea → route → author spec → spec PR, repeated.
- **Route** an idea to a registered project (9.2), human-confirmed; **create** a new project
  (9.2 `create`) when none fits.
- **Author** the spec by reusing the existing DECIDE skills (brainstorm→stories→plan) in the
  target repo, on a `spec/<feature>` branch, opened as a **PR** — the human's merge is approval.
- **Close the flywheel:** read the 9.1 brain store at plan time and surface relevant prior
  lessons (kickbacks/halts/retry-hotspots/narratives) into planning.
- **Report** aggregate token spend + kickback/halt/retry rates (read-only governor).

**Non-Goals**
- **The brain does NOT *manage* daemons** — daemons stay independent and fault-isolated. *(Revised
  per ADR-005, 2026-06-25: the brain MAY **launch** a project's daemon as a detached, fire-and-forget
  convenience, but holds no control connection/supervision/lifecycle ownership. A launched daemon
  still builds human-merged specs only, so launching creates no autonomous build path.)*
- **No build, no auto-merge** — the human approval gate (spec PR merge) is non-negotiable.
- No governor *broker*/gating (per-daemon self-limit stands); no UI (9.4); no multi-host.
- No auto-applied harness self-edits (propose-only PRs through existing gates).

## Users / Personas

- **The human operator** (James) — brings feature ideas, confirms routing, reviews/merges spec
  PRs. The brain is their planning co-pilot across all projects.
- **The per-project daemon** — the downstream consumer of the merged specs (unchanged).

## Functional Requirements

- **FR-1:** `conduct brain` starts a **long-lived interactive loop**; on start it loads the
  registry (9.2 `RegistryReader`) for known projects and opens the brain store (9.1) for
  cross-project memory.
- **FR-2:** Each iteration accepts a **feature idea** (free text); the loop continues until the
  operator exits. Context persists across ideas in a session.
- **FR-3:** For an idea, the brain **proposes a target project** (inferred from the idea +
  registry) and the operator **confirms or redirects** before any authoring happens.
- **FR-4:** *(edge)* If **no registered project fits**, the brain offers to **create** one via the
  9.2 `create` command (scaffold + register) and routes the idea there.
- **FR-5:** Before/at planning, the brain **reads the 9.1 brain store** and surfaces the
  **relevant** prior lessons (kickbacks/halts/retry-hotspots + narrative refs) — for the target
  project and similar features — injecting them into the planning context (closes the flywheel).
- **FR-6:** The brain **authors the spec by orchestrating the existing DECIDE skills**
  (brainstorm→stories→plan) against the **target project's repo**, on a `spec/<feature>` branch —
  producing the same PRD/stories/plan artifacts a normal DECIDE phase would.
- **FR-7:** The brain **opens a PR** containing the authored spec in the target project; the
  operator's **merge is the approval**. The brain **never builds** and **never auto-merges**.
- **FR-8:** *(revised per ADR-005)* The brain **does not *manage* daemons** — it may **launch** one
  detached (fire-and-forget) as a convenience, but holds no control connection/supervision/lifecycle
  ownership. After the spec PR merges, the project's independent daemon consumes it (building merged
  specs only). The brain takes no further action on that feature.
- **FR-9:** The brain **reports** aggregate token spend and kickback/halt/retry rates read from
  the brain store (read-only) — it does **not** gate or broker execution.
- **FR-10:** *(edge, non-negotiable)* The human approval gate holds: **no build proceeds without a
  merged spec PR**; the brain never auto-merges; any harness self-edit it might propose is a
  PR through the existing validation/no-auto-merge gates, never auto-applied.
- **FR-11:** *(edge)* The brain operates across **separate project repos** (9.2 isolation):
  authoring for project A creates the branch/PR **only** in A's repo and never touches B.
- **FR-12:** *(measurability)* The flywheel is measurable: the brain can compute whether
  kickback/halt/retry rates **fall across successive brain-planned features** (per 9.1's metric),
  so we can tell it's learning, not accumulating noise.

## Non-Functional Requirements

- **Reuse over reinvention:** routing/authoring reuse the registry reader (9.2), the brain store
  (9.1), the DECIDE skills, and the existing PR machinery — no parallel planning stack.
- **Non-autonomy by construction:** every path to "a daemon builds" passes through a human-merged
  spec PR. There is no code path from brain → build without that merge.
- **Cross-repo safety:** the brain resolves the target repo from the registry's canonical path and
  confines all writes (branch/PR) to it.
- **Read-only over the store/registry where it consumes** (it writes specs to project repos, not
  to the store/registry — except creating a project via 9.2, which registers).

## Acceptance Criteria / Success Metrics

- Every FR covered by tests (vitest + real git repos for the cross-repo authoring/PR paths):
  1. `conduct brain` loop starts, loads registry + store, accepts an idea, loops, exits cleanly.
  2. Routing proposes a project from registry+idea and requires confirmation; redirect works;
     no-fit → offers `create`.
  3. Flywheel: given seeded store signals, the brain surfaces the relevant prior lessons for the
     target project into the planning context.
  4. Authoring runs brainstorm→stories→plan in the **target** repo on a `spec/<feature>` branch
     and opens a PR; **no build runs**; **no auto-merge**.
  5. Cross-repo isolation: authoring for project A leaves project B untouched.
  6. Governor reporting computes aggregate spend + kickback/halt/retry rates from the store.
  7. Non-negotiable gate: there is no path from idea → build without a merged spec PR.
- Existing tests still pass; harness integrity green; CHANGELOG `[Unreleased]` + README/conductor
  README updated; any new skill/mode reflected in HARNESS.md.

## Scope

### In Scope
- The `conduct brain` interactive mode (loop, idea intake, exit).
- Routing (registry-informed, human-confirmed) + create-on-no-fit (9.2).
- Flywheel-memory read (9.1 store → relevant lessons → planning context).
- Spec authoring via DECIDE skills in the target repo + spec-branch+PR handoff.
- Read-only governor reporting. Tests + docs.

### Out of Scope
- Spawning/managing daemons; build; auto-merge; governor broker/gating; UI (9.4); multi-host;
  auto-applied harness self-edits.

## Key Decisions & Rationale

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Brain is a **planning/routing service**, not a daemon parent | Locked program decision; daemons stay independent (fault isolation); coupling via repo+registry. |
| 2 | **Reuse the DECIDE skills** to author specs | Consistency + dogfooding; the operator reviews the same PRD/stories/plan shapes; no second planning path. |
| 3 | **Spec → branch + PR; merge = approval** | The non-negotiable human gate, async/phone-friendly, full audit trail; the brain never builds/merges. |
| 4 | **Interactive `conduct brain` loop** | "Brain loops for each feature idea" with persistent cross-idea context. |
| 5 | **Flywheel read in 9.3** (full scope) | It's the whole point of 9.1; the brain reading lessons at plan time is what makes the loop a flywheel, not a diary. |
| 6 | **Read-only governor** (report, don't gate) | Per-daemon self-limit suffices at solo scale; a broker is unjustified (deferred unless contention is observed). |
