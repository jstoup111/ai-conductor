# PRD: Phase 9.0 — Daemon Rebase-on-Latest + Conflict→HALT

**Date:** 2026-06-25
**Status:** Approved
**Predecessor design:** `.docs/decisions/supervisor-brain-followup.md` §3
**Program:** Phase 9 (Supervisor/Brain). This is the first, independently-shippable slice;
the brain stack (9.1–9.4) is out of scope here.

## Problem / Background

The daemon forks a worktree from the base branch at spin-up and **never reconciles it**. A
long build races a base branch that other merged PRs have advanced, so the feature branch goes
stale: its PR conflicts at human-merge time, or it was built against a base that is missing a
dependency it needs.

Scenario D (concurrent features) passed in Phase 7 validation only because its two features
touched **disjoint** code. The moment two real features touch overlapping code, this bites. It
is a **daemon-correctness** issue, independent of the brain — so it ships first, on its own,
before any daemon runs on real overlapping features.

The fix reuses the loop's existing machinery: **"the base branch advanced" is just an external
invalidation, the same shape as a kickback.** The gate loop already re-verifies on a kickback,
and already has a `.pipeline/HALT` park-for-human pattern. No new control-flow paradigm is
introduced.

## Goals & Non-Goals

**Goals**
- The daemon always hands a human a **clean, rebased, still-green PR** — never a branch that
  conflicts at merge time or was built against a stale base.
- Preserve **never-auto-merge** and **never auto-resolve a non-trivial merge** — the only
  conflict ever resolved automatically is the known-safe CHANGELOG `[Unreleased]` case.
- Reuse the existing verdict/selector kickback machinery and HALT park-for-human pattern
  rather than inventing parallel control flow.

**Non-Goals**
- No brain, registry, supervisor mode, cross-project governor, or UI (Phases 9.1–9.4).
- No opportunistic mid-build rebasing — rebase happens **once**, before `finish`.
- No smart conflict resolution beyond the single known CHANGELOG case.
- No change to how features are discovered or how PRs are opened beyond inserting the rebase.

## Users / Personas

- **The human operator** who runs per-project daemons and later merges the PRs they produce.
  They want every daemon PR to merge cleanly with no surprise conflicts, and to be cleanly
  parked (not silently broken) when a conflict genuinely needs their judgment.
- **The daemon itself** (automated consumer) — needs a deterministic, resumable rebase step
  that fits the existing gate loop.

## Functional Requirements

- **FR-1:** After the `build` and `manual_test` gates are satisfied and **before** the
  `finish` step runs, the daemon rebases the worktree's feature branch onto the latest base.
- **FR-2:** The rebase base is **origin's default branch**, discovered dynamically
  (`git symbolic-ref refs/remotes/origin/HEAD`), and fetched first (`git fetch origin
  <default>`). The base branch name is never hardcoded to `main`/`master`.
- **FR-3:** *(edge)* If there is no `origin` remote, or the fetch/default-branch discovery
  fails, the daemon falls back to rebasing onto the **local base branch** and proceeds; lack
  of a remote never fails the feature.
- **FR-4:** *(edge)* If the feature branch is already up to date with the base (the rebase is a
  no-op — no new base commits), re-verification is **skipped** and the loop proceeds directly
  to `finish`.
- **FR-5:** If the rebase applies cleanly **and changes the worktree tree** (post-rebase tree
  differs from pre-rebase), the daemon invalidates the downstream loop gates (`build`, and
  `manual_test` if it ran) via the **same verdict mechanism used for kickbacks**, forcing
  re-verification before `finish`.
- **FR-6:** Re-verification after a file-changing clean rebase runs **through the existing
  gate-loop machinery** (retry / autoheal / anti-oscillation). If it self-heals, the loop
  proceeds to `finish`/PR. If the loop becomes stuck (existing `MAX_GATE_SELECTIONS`
  threshold), it writes `.pipeline/HALT` via the existing path. There is **no special-case
  immediate-HALT** for post-rebase gate failure.
- **FR-7:** On a **CHANGELOG.md** conflict during rebase, the daemon auto-resolves it: it takes
  the base/upstream version of the file (which already contains other features' merged
  `[Unreleased]` entries) and re-appends **only this feature's own** `[Unreleased]` additions
  (captured from `base..HEAD` before the rebase), without duplicating blocks, then continues
  the rebase. (Note the rebase ours/theirs inversion: during rebase, "ours" is the upstream
  base being replayed onto.)
- **FR-8:** *(edge)* On **any non-CHANGELOG conflict** — or a CHANGELOG conflict the
  auto-resolver cannot safely handle — the daemon writes `.pipeline/HALT` with a note listing
  the conflicted files and the resume procedure, **keeps the worktree**, and **leaves the
  rebase paused in its conflicted state** for the human. No non-trivial merge is ever
  auto-resolved.
- **FR-9:** *(edge)* A rebase HALT keeps the worktree (park-for-human), does **not** mark the
  feature processed, and does **not** open a PR. After the human resolves the conflict,
  `git rebase --continue`s, clears `.pipeline/HALT`, and re-queues, re-invoking the daemon
  reuses the worktree, finds the rebase a no-op (FR-4), re-verifies if needed, and opens the
  PR. The rebase step is idempotent and resumable.
- **FR-10:** The rebase step emits structured events to the existing event log (at minimum a
  rebase-outcome event and the HALT reason) so `conduct --report` — and the future 9.1 retro
  signal — can observe rebase outcomes (clean / changed-files / changelog-auto-resolved /
  conflict-halt).

## Non-Functional Requirements

- **Reuse over reinvention:** invalidation must go through the existing verdict/selector
  kickback path; HALT must go through the existing park-for-human path. No parallel control
  flow.
- **Idempotent & resumable:** re-running the daemon on a parked or partially-rebased worktree
  must be safe and must converge (FR-9).
- **No hard network dependency:** the feature must work in repos with no remote (FR-3),
  including isolated test fixtures.
- **Safety invariants preserved:** never-auto-merge; never auto-resolve a non-trivial conflict.

## Acceptance Criteria / Success Metrics

- Every FR is covered by passing tests (vitest, matching the existing
  `test/integration/gate-loop.test.ts` patterns). Required cases:
  1. Clean **no-op** rebase (base unchanged) → straight to `finish`, no re-verify (FR-4).
  2. Clean **file-changing** rebase → build/manual_test invalidated → re-verified → `finish`
     (FR-5, FR-6).
  3. **CHANGELOG** conflict → auto-resolved (base entries + this feature's lines, no
     duplication) → rebase continues → `finish` (FR-7).
  4. **Non-CHANGELOG** conflict → `.pipeline/HALT` written, worktree kept, no PR, rebase left
     paused (FR-8, FR-9).
  5. **No-remote** repo → falls back to local base, proceeds (FR-3).
  6. **Post-rebase gate failure** → re-verifies via the normal loop; stuck loop → HALT (FR-6).
  7. **Resume**: after a simulated human resolve + clear-HALT + re-queue, the daemon converges
     to a PR (FR-9).
- Existing daemon/loop tests (Phase 7 scenarios C single-feature and D concurrent) still pass.
- Repo validation passes: `test/test_harness_integrity.sh` green; `CHANGELOG.md`
  `[Unreleased]` updated; `README.md` and `src/conductor/README.md` updated to document the
  rebase-before-finish behavior (docs-track-features).

## Scope

### In Scope
- A rebase-before-`finish` step in the daemon/loop tail (insertion mechanism decided at
  architecture-review: new pseudo-gate vs. inline in `advanceTail`).
- Base discovery + fetch with local fallback.
- File-change detection driving kickback-style invalidation of `build`/`manual_test`.
- The CHANGELOG `[Unreleased]` auto-resolver.
- Conflict→HALT with worktree kept + rebase left paused + resume procedure.
- Structured rebase events.
- Tests for all FRs + docs updates.

### Out of Scope
- Brain / registry / supervisor mode / cross-project governor / UI (9.1–9.4).
- Opportunistic mid-build rebasing.
- Any conflict auto-resolution beyond the CHANGELOG case.
- Changes to backlog discovery or PR-body generation.

## Key Decisions & Rationale

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Rebase **once before `finish`**, after build+manual_test green | Simplest cadence that closes the staleness gap; the doc's stated starting point. Mid-build rebasing adds churn for little gain. |
| 2 | Base = **origin's default branch**, discovered + fetched | Sibling features only land on the **remote** (never-auto-merge), so the local base goes stale; only `origin/<default>` reflects merged sibling PRs. Discovery (not hardcoding) per the default-branch-discovery convention. |
| 3 | **Local-base fallback** when no remote | Keeps the feature working in remote-less repos and test fixtures; absence of a remote is not a failure. |
| 4 | Clean file-changing rebase → **kickback-shaped invalidation** → re-verify | "Base advanced" is the same shape as an external invalidation; reuse the validated verdict/selector path instead of new control flow. |
| 5 | Post-rebase failure → **normal loop**, HALT only if stuck | Maximally reuses the kickback re-verify path; lets simple post-rebase breaks self-heal; fewer human interrupts. |
| 6 | **Auto-resolve CHANGELOG only**; HALT all other conflicts | CHANGELOG `[Unreleased]` collisions are the known, expected parallel-worktree conflict with a safe resolution; everything else needs human judgment (never auto-resolve non-trivial merges). |
| 7 | Conflict HALT **leaves the rebase paused** in-place (not aborted) | Preserves exact conflict context for the human to resolve; the parked worktree won't be touched by the daemon until HALT is cleared, so the fragile mid-rebase state is safe; re-invocation after `--continue` converges (FR-9). |

## Dependencies

- Existing gate-loop verdict/selector machinery (`gate-verdicts.ts`, `selector.ts`,
  `conductor.ts` `advanceTail`).
- Existing `.pipeline/HALT` park-for-human pattern and worktree-keep teardown
  (`daemon-runner.ts`, `daemon-deps.ts`).
- Worktree lifecycle in `daemon-deps.ts` `createWorktree`.
- `git` (fetch/rebase) and the existing event log (`events.jsonl`).

## Open Questions

- None blocking. The **insertion mechanism** (a new pseudo-gate step in the registry vs.
  inlining the rebase in `advanceTail` before `finish`) is deferred to architecture-review,
  which will weigh it against the Phase 8 "derive topology from the step registry" work.
- Whether `manual_test` actually re-runs on invalidation depends on its per-feature auto-skip
  status; the kickback machinery handles either case — confirmed at plan time, not a blocker.
