# PRD: Phase 9.1 — Structured Retro Signal + Brain Memory Store

**Date:** 2026-06-25
**Status:** Approved
**Program:** Phase 9 (Supervisor/Brain). Second slice — the flywheel *plumbing*. Builds on 9.0.
**Predecessor design:** `.docs/decisions/supervisor-brain-followup.md` §4.1, §7

## Problem / Background

The daemon ships features but its hard-won signals — kickbacks (with reasons), HALTs (gate +
why), retry hotspots, `--report` telemetry (durations, token spend), and the retro's
human-judgment narrative — evaporate into each project's repo or are lost. The flywheel
(ship → retro → brain learns → better plans) needs those signals **accumulated in one
cross-project place the future brain (9.3) can read**. A retro that never re-enters a future
plan is just a diary.

9.1 builds the **emission + store** side now (the signals already exist per-feature). The
**consumption** side ("brain reads memory at plan time") is deferred to 9.3, when the brain
exists — so 9.1 is shippable and testable in isolation.

A locked steer shapes this: **harness/conductor retros must not live in the daemon's project
repo.** They belong in the brain's running state. So daemon-built features stop writing
`.docs/retros/` into their repos; that content goes to the brain store instead.

## Goals & Non-Goals

**Goals**
- On daemon feature completion, persist a **structured, aggregable signal** + the **retro
  narrative** into a cross-project **brain store**, keyed by project/feature.
- Keep daemon retros **out of project repos**.
- Make the store shaped so kickback/halt/retry rates are **measurable across features/projects**
  (the learning metric that tells us the brain is learning, not accumulating noise).

**Non-Goals**
- **No consumer/reader** — no "brain reads at plan time", no brain, no registry, no UI, no
  cross-project aggregation *tool*. The store merely *enables* later aggregation (9.3+).
- **No emission from manual `/conduct` runs** — daemon runs only.
- No change to *what* signals exist (events.jsonl, `--report`) — only their collection + routing.

## Users / Personas

- **The future brain (9.3)** — the eventual reader; 9.1 produces the store it will consume.
- **The human operator** — can inspect the store to see cross-project trends before the brain
  exists (CLI-grep over JSONL); benefits from clean project repos (no retro clutter).
- **The daemon** — the autonomous producer that emits on each feature completion.

## Functional Requirements

- **FR-1:** On **daemon** feature completion — outcome `done` **or** `halted` — the daemon emits
  exactly **one** structured signal record to the brain store. Manual `/conduct` runs emit
  nothing.
- **FR-2:** The brain store lives at **`~/.ai-conductor/brain/`** (parallel to the harness user
  config `~/.ai-conductor/config.yml`), **overridable** via env/config (e.g.
  `$AI_CONDUCTOR_BRAIN_DIR`), and is created if absent. It is **never** inside a project repo.
- **FR-3:** Each signal is appended as one JSON line to a signals log
  (`~/.ai-conductor/brain/signals.jsonl`) with a versioned schema:
  `{schemaVersion, ts, project, feature, runId, outcome, kickbacks[], halts[], retryHotspots[],
  tokens{input,output,cacheRead,cacheCreation}, durationByStep{}, narrativeRef}`.
  (Exact field types locked at architecture/ADR.)
- **FR-4:** The structured fields are **assembled from existing sources** — the feature's
  `events.jsonl` (kickbacks, `loop_halt`, `step_completed` tokenUsage, retries, `rebase_*`) +
  `daemon-runner` `FeatureOutcome` (outcome, costTokens, halt reason) + `report-renderer`
  aggregation (durations, retries, token spend). No new instrumentation of the loop.
- **FR-5:** For a **done** feature, the daemon writes the **full retro narrative** to the brain
  store (`~/.ai-conductor/brain/narratives/<project>/<feature>.md`), **not** to the project
  repo's `.docs/retros/`; `narrativeRef` points to it.
- **FR-6:** For a **halted** feature, the daemon writes a **short halt narrative** (why it
  halted — the failure story, not a full feature retro) to the brain store narratives dir,
  referenced by `narrativeRef`.
- **FR-7:** *(edge)* Daemon-built features write **no retro output into their project repo**
  (`.docs/retros/` stays empty for daemon runs). Manual runs are unchanged (still write to the
  repo).
- **FR-8:** *(edge)* Re-emitting on a daemon **re-run** of the same feature does not corrupt or
  silently overwrite history: records carry a `runId`; a re-run appends a **new** record (new
  `runId`) and a versioned narrative, so prior signals are retained. (Dedup/run-id strategy
  locked at architecture.)
- **FR-9:** The stored fields are sufficient and typed so **kickback / halt / retry rates** can
  be computed across successive features and projects (the measurability metric) by a later
  reader — verified by computing those rates from a set of fixture records in tests.
- **FR-10:** *(edge)* Emission is **best-effort**: a brain-store write failure (unwritable dir,
  disk error) is **logged and swallowed** — it must never fail or block the daemon's feature
  completion, teardown, or PR. The feature still ships.
- **FR-11:** *(edge)* The signals log is **append-safe under concurrency**: the daemon runs
  features in parallel (`--concurrency > 1`), so concurrent emissions must not interleave/corrupt
  JSON lines (atomic whole-record append).

## Non-Functional Requirements

- **Reuse over reinvention:** assemble from `events.jsonl` + `report-renderer` + `FeatureOutcome`;
  do not re-instrument the loop.
- **Isolation:** no dependency on a brain/registry/UI; the store is a passive sink.
- **Token/size discipline:** the signals log stays compact (one line/feature); full narratives
  live as separate files referenced by `narrativeRef`, not inlined.
- **Forward-compatibility:** `schemaVersion` on every record so the 9.3 reader can evolve.

## Acceptance Criteria / Success Metrics

- Every FR covered by passing tests (vitest, matching existing daemon/engine test patterns):
  1. Daemon `done` feature → one signal line with correct fields + a full-retro narrative file in
     the store; **nothing** in the repo's `.docs/retros/`.
  2. Daemon `halted` feature → one signal line (outcome=halted, halt gate+reason) + a short halt
     narrative file.
  3. Manual `/conduct` run → **no** store emission; repo retro behavior unchanged.
  4. Store path defaults to `~/.ai-conductor/brain/` and honors the override env/config.
  5. Re-run of a feature → second record retained (new `runId`), first not clobbered.
  6. Kickback/halt/retry rates computable from fixture records (measurability).
  7. Unwritable store dir → emission logged + swallowed, feature still completes/ships.
  8. Concurrent emissions (≥2 workers) → no corrupted JSONL lines.
- Existing daemon/loop tests still pass; harness integrity green; CHANGELOG `[Unreleased]` +
  README/`src/conductor/README.md` updated (docs-track-features).

## Scope

### In Scope
- The brain store (dir layout, default path + override, append-safe JSONL, narratives dir).
- Signal assembly from existing sources + the record schema.
- Emission on daemon feature completion (done + halted), best-effort + concurrency-safe.
- Redirect of daemon retro/halt narratives to the store (off the project repo).
- Tests + docs.

### Out of Scope
- Any reader/consumer, the brain, registry, supervisor mode, UI, cross-project aggregation tool
  (9.2–9.4).
- Emission from manual `/conduct` runs.
- New loop instrumentation or new event types beyond what 9.0 already emits.

## Key Decisions & Rationale

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Emission + store **only**; consumption deferred to 9.3 | The brain doesn't exist yet; the signals do. Ship the plumbing, keep it testable in isolation. |
| 2 | Store at **`~/.ai-conductor/brain/`** | That's where the harness's user-level config already lives (`user-config.ts`, `bin/install`); the brain store belongs beside it, not in any project repo. |
| 3 | **Daemon runs only** emit | The autonomous flywheel the doc describes; manual runs are human-driven and would add noise. |
| 4 | Daemon retros go to the **store, not the repo** | Locked steer: conductor retros must not clutter the daemon's project repo; they're the brain's running state. |
| 5 | Compact signal line + **separate narrative file** (`narrativeRef`) | Keeps the aggregable index light; full text one hop away. Token-efficient. |
| 6 | Halted features get a **short halt narrative** (not a full retro) | A halted feature has no finished work to retro on, but the *why-it-halted* story is high-value halt-pattern signal for the brain. |
| 7 | Emission **best-effort + concurrency-safe** | Never let a learning-signal write break a real feature ship; the daemon runs features in parallel. |

## Dependencies

- 9.0-era signal sources: `events.jsonl` / `ConductorEvent` (incl. `rebase_*`), `report-renderer.ts`,
  `daemon-runner.ts` `FeatureOutcome`, the retro step.
- Harness user-config conventions (`~/.ai-conductor/`, `user-config.ts`).
- `git`/fs; no new external deps.

## Open Questions

- None blocking. The **record schema field types**, the **append-safe write mechanism**, and the
  **re-run dedup/run-id** strategy are architecture/ADR decisions (analogous to 9.0's
  insertion-mechanism ADR), to be locked in architecture-review.
- Whether the 9.3 reader contract should be sketched now as a stub interface (so 9.1's schema is
  consumer-aware) — lean yes, but the stub ships no behavior.
