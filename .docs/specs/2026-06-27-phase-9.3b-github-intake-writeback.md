# Phase 9.3b — GitHub-Issues Intake Adapter + Bidirectional Write-Back

**Status:** Approved
**Date:** 2026-06-27
**Author:** James Stoup (with Claude)
**Supersedes:** none (additive follow-on to `2026-06-26-phase-9.3-engineer-redesign.md`)
**Phase:** 9.3b (deferred half of the 9.3 engineer redesign)

## Problem / Background

Phase 9.3 redesigned the engineer as an agent-hosted idea→spec loop and **locked an intake
port + `Envelope` contract**, wiring only the synchronous **`claude-session` (chat)** adapter.
Three capabilities were deliberately deferred to 9.3b because they are only meaningful together
and add no value without an external source:

1. a **`github-issues` intake adapter** (an async, pull-based source),
2. an **inbox poll / async capture buffer** (the capture/processing split the chat path didn't need), and
3. **bidirectional write-back** (`IntakePort.report()`, currently a no-op).

The 9.3 seam was explicitly designed so 9.3b is **purely additive**: new adapter + queue behind
interfaces + config, with **zero changes to routing, DECIDE, or the daemon liveness/lock code**.

The code already in place (merged via #85) that 9.3b extends:
- `intake/port.ts` — `Envelope { id, source, sourceRef, text, hintRepo?, status, receivedAt }`,
  `EnvelopeStatus = pending|routed|deciding|done`, and `IntakePort.report(sourceRef, status)`
  (reserved, no-op).
- `intake/idempotency.ts` — an **in-memory** dedup guard keyed by `source\x00sourceRef`
  (resets every process — insufficient across sessions).
- `intake/claude-session.ts` — the synchronous chat adapter (`report()` is a no-op).
- `engine/daemon-lock.ts` — liveness / `O_EXCL` pidfile mutex. **Out of bounds for 9.3b.**

**Why async matters here:** the chat adapter captures and processes an idea in the same
in-session step. A GitHub issue arrives with **no operator present**, yet routing and DECIDE are
**interactive and human-gated** and cannot run unattended. So 9.3b must split *capture* (produce
an Envelope) from *processing* (route → DECIDE → spec PR), with a durable buffer between them.

## Goals & Non-Goals

**Goals**
- Add a `github-issues` adapter that captures issues **assigned to the operator across all
  registered repos** as Envelopes, through the existing port contract.
- Persist captured Envelopes in a durable on-disk inbox so async-sourced work survives until an
  interactive engineer session processes it.
- Guarantee **exactly-once pull**: an issue is captured once and never re-ingested while it sits
  open, via a durable ledger plus a GitHub-side marker.
- Realize **write-back**: post lifecycle progress back to the originating issue.
- Auto re-enable an issue whose spec PR was **closed unmerged**, with a churn guard.
- Shape the buffer behind a swappable **`IntakeQueue`** interface so a future distributed worker
  pool is a drop-in backend — without building distributed infrastructure now.

**Non-Goals**
- No distributed queue backend (shared lease store / visibility timeout / multi-worker claim).
  Designed-for, not built. Likely a later phase.
- No always-on background daemon or harness-supervised cron. The harness exposes a poll
  command; the operator may wire it into cron themselves.
- No other sources (email, Slack) — the interface accommodates them; this phase ships github only.
- **No changes to FR-20 / daemon execution topology / the `O_EXCL` liveness lock.**
- No auto-closing or state-transitioning of issues beyond the handled marker. Issue closure
  belongs to the build/merge flow, not the engineer (a spec PR being *opened* is not "done"
  for the issue).
- No changes to routing or the DECIDE skills.

## Users / Personas

- **The operator (solo dev, phone-friendly):** files ideas as GitHub issues assigned to
  themselves from anywhere, then later runs an engineer session that drains the queue one idea at
  a time. Gets progress comments back on each issue without watching a terminal.

## Functional Requirements

FR numbering continues from the 9.3 PRD (which ended at FR-24) to keep cross-phase references
unambiguous.

### A. The `github-issues` capture source

- **FR-25:** Define an async **capture interface** (e.g. `IntakeSource.poll(): Promise<Envelope[]>`)
  distinct from `IntakePort.report()`. The engineer core depends only on the interface (loose
  coupling, mirroring FR-13). The `claude-session` adapter remains synchronous and is **not** an
  `IntakeSource`.
- **FR-26:** The github-issues poll fetches issues **assigned to the operator** (`gh issue list
  --assignee @me --state open`) across **all repos in the 9.2 registry**, producing one Envelope
  per issue: `source = "github-issues"`, `sourceRef = "<owner>/<repo>#<number>"`, `text` = issue
  title + body, `hintRepo` = the repo the issue was filed in (a routing hint, not a routing
  decision).
- **FR-27 (negative, auth/availability):** if `gh` is unauthenticated/unavailable, or a specific
  repo is inaccessible, that source/repo **fails with a clear error and captures nothing for it**
  (no partial or blank Envelopes); other repos still poll, and the engineer launch never crashes.
- **FR-28 (edge, empty):** an issue whose title **and** body are empty/whitespace yields **no**
  Envelope (reuse `EmptyEnvelopeTextError` semantics) and is logged, not silently enqueued blank.

### B. Async inbox buffer + swappable queue seam

- **FR-29:** A **durable on-disk inbox** persists captured Envelopes across sessions. All buffer
  access goes through a narrow swappable **`IntakeQueue`** interface (`enqueue` / `claim` /
  `ack` / `release`) so a future distributed-pool backend is a drop-in implementation with **zero**
  changes to the adapter or the engineer loop.
- **FR-30 (claim / concurrency):** processing **atomically claims** the oldest pending Envelope
  (`pending → claimed`) before routing, so two concurrent engineer sessions never process the same
  Envelope. Successful spec delivery **acks** it (`claimed → done`); a crash/abandon **releases**
  the claim so it is reclaimable (no Envelope is ever permanently stuck). This claim is wholly
  independent of the daemon's `O_EXCL` lock — **FR-20 is untouched**.
- **FR-31:** `conduct-ts engineer` at launch: poll all configured sources → ledger/idempotency
  filter → enqueue new Envelopes → process the **oldest** claimable Envelope this session (one
  idea per session, unchanged). If the inbox is empty after polling, fall back to the existing
  interactive `claude-session` chat capture.
- **FR-32:** A standalone **`conduct-ts engineer poll`** subcommand runs **capture-and-enqueue
  only** (no processing), writing to the same inbox. This is the seam an operator may cron later;
  the harness introduces and supervises **no** always-on process.

### C. Durable sequencing / idempotent pull

- **FR-33:** A **durable intake ledger** keyed by `source + sourceRef` records lifecycle state
  (`unseen → pending → claimed → routed → deciding → done`) plus `{ branch, prUrl, attempts,
  timestamps }`. It survives process exit (the in-memory idempotency guard does not, and is
  insufficient on its own).
- **FR-34 (idempotent pull):** poll **skips** any candidate already present in the ledger in a
  non-resettable state. An issue is captured **exactly once** regardless of how many times it is
  polled, even while it remains open + assigned on GitHub.
- **FR-35 (distributed-ready dedup anchor):** on reaching `done`, the adapter applies a GitHub-side
  **`engineer:handled`** label to the issue; poll **also** skips issues bearing it. The ledger is
  the authoritative local guard; the label is the **globally visible** anchor a future multi-worker
  pool can read. The label is an **output** marker only — it does not re-introduce a trigger label
  on intake (intake remains assignee-based).

### D. Bidirectional write-back (`report()` realized)

- **FR-36:** the github-issues adapter's `report(sourceRef, status, meta?)` posts a progress
  **comment** on the originating issue at lifecycle transitions: `routed` → "Routed to `<repo>`";
  `done` → "Spec PR opened: `<url>`" (and applies the `engineer:handled` label per FR-35). The
  `claude-session` adapter's `report()` stays a no-op.
- **FR-37 (negative, non-fatal):** write-back failure (issue deleted, network, perms) is **logged
  and non-fatal** — it never blocks or reverts spec delivery (the spec PR is the real artifact;
  the comment is advisory). The local ledger transition is still recorded.
- **FR-38 (idempotent write-back):** re-invoking `report()` for the same `(sourceRef, status)`
  does **not** post a duplicate comment or re-apply the label (check-before-write).

### E. Re-eligibility on a rejected spec

- **FR-39:** poll inspects each `done` ledger entry's recorded spec `prUrl`
  (`gh pr view --json state,mergedAt`); **CLOSED + not merged → reset** the ledger entry to
  `unseen`, strip the `engineer:handled` label, increment `attempts`, so the issue re-enters the
  inbox automatically. A **MERGED** spec PR is terminal and never reopened.
- **FR-40 (churn guard):** after **2** auto-reopens for the same `sourceRef`, stop auto-reopening,
  mark the entry `needs-manual`, and emit a warning. `conduct-ts engineer forget <sourceRef>`
  manually clears any ledger entry (and strips the label) to re-enable it.

## Non-Functional Requirements

- **Additivity (hard):** zero changes to routing, the DECIDE skills, or the daemon
  liveness/`O_EXCL` lock. The engineer core depends only on interfaces.
- **Loose coupling:** new source behind `IntakeSource`, buffer behind `IntakeQueue`; a distributed
  pool is a swap of the queue implementation only.
- **Crash safety:** claimed Envelopes are reclaimable after a crash; no permanently stuck items;
  the ledger is durable and append-safe.
- **End-to-end idempotency:** pull once (FR-34/35), write back once (FR-38).
- **Cross-repo safety:** capture reads many repos (read-only via `gh`), but authored artifacts and
  the spec branch/PR still land **only** in the routed target repo (existing `AuthoringGuard`).
- **Read-only over registry/store** except the already-sanctioned project-create path.

## Acceptance Criteria / Success Metrics

- Every FR covered by **vitest**; all `gh` interactions injected/faked (no live network in unit
  tests).
- An integration test over temp repos + a fake `gh` exercises the full async path:
  poll → enqueue → claim → route(stub) → `report()` → done, including the **idempotent re-poll**
  (second poll captures nothing) and the **auto-reopen** path (closed-unmerged PR → re-enters;
  third reject hits the churn guard).
- Manual test: file an issue assigned to self → `conduct-ts engineer` drains it → progress
  comments appear on the issue → `engineer:handled` label applied → re-running poll is a no-op.

## Scope

**In**
- `github-issues` capture adapter (assignee-based, all registered repos).
- Poll-on-launch + standalone `conduct-ts engineer poll` subcommand.
- Durable inbox + ledger behind the `IntakeQueue` / `IntakeSource` interfaces.
- Write-back comments + `engineer:handled` label via realized `report()`.
- Auto-reopen on closed-unmerged spec PR with a 2-attempt churn guard + `engineer forget`.

**Out (→ future / other phases)**
- Distributed queue backend (shared lease store, visibility timeout, multi-worker claim).
- Always-on background poller/daemon supervised by the harness.
- Email / Slack / other sources.
- Any change to FR-20, daemon topology, or the `O_EXCL` lock.
- Auto-closing or transitioning issues beyond the handled label.

## Key Decisions & Rationale

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | **Assignee-based intake**, all registered repos | Lowest friction to file from anywhere; no trigger-label discipline. Dedup is harness-side (no label to toggle on intake). |
| 2 | **Poll-on-launch**, no always-on poller | Processing is human-gated, so for **re-queryable pull sources** capture latency is irrelevant — the issue waits on GitHub either way. A background poller adds a supervised process + concurrency surface for **zero** throughput gain at solo scale. |
| 3 | **Durable ledger + GitHub `engineer:handled` label** as dual dedup anchor | Ledger is the local authority; the label is the one piece of dedup state a **future distributed worker pool** can see natively across machines. |
| 4 | **`IntakeQueue` seam now, file-backed impl** | Future distributed pool = swap the backend, no adapter/loop changes — mirroring how 9.3's port made 9.3b additive. Avoids building distributed infra prematurely. |
| 5 | **Auto-reopen on closed-unmerged PR, capped at 2** | Matches the operator's iterate-on-reject workflow without an infinite reject→re-ingest loop; merged is terminal; `forget` is the manual escape hatch. |
| 6 | **Write-back is advisory, non-fatal** | The spec PR is the real artifact; a failed comment must never block or revert delivery. |

## Dependencies

- 9.3 intake port / `Envelope` / idempotency guard (merged #85) — extended, not modified.
- 9.2 registry reader (the repo set to poll).
- `gh` CLI, authenticated, with issue read + comment + label scopes.
- Existing PR machinery + `AuthoringGuard` (artifacts confined to the routed target repo).

## Resolved Decisions (formerly Open Questions)

1. **Marker form → label.** The handled marker is the `engineer:handled` **label** (queryable,
   visible). The adapter ensures the label exists in the target repo (auto-create if missing).
2. **`text` composition → title + body.** The Envelope carries the issue title + body; DECIDE
   summarizes downstream.
3. **Comment verbosity → both.** Write-back posts a comment at **both** `routed` and `done`
   transitions.
