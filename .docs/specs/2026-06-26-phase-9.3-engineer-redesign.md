# PRD: Phase 9.3 (Redesign) — Engineer Loop, Intake Port & Daemon Liveness

**Date:** 2026-06-26
**Status:** Approved
**Program:** Phase 9 (Supervisor/Engineer). Redesign of the shipped 9.3 — **supersedes**
`.docs/specs/SUPERSEDED-2026-06-25-phase-9.3-supervisor-engineer.md`. Based on `main`
(incl. 9.0/9.1/9.2) on branch `feature/phase-9-3-supervisor-brain` (PR #85).
**Handoff of record:** `.handoffs/phase-9.3-engineer-redesign-DECIDE.md`

## Problem / Background

The shipped 9.3 **diverged from its own PRD**. PRD **FR-6** required the engineer to *author specs
by orchestrating the existing DECIDE skills* (brainstorm→stories→plan), human-gated. The shipped
`engine/engineer/loop.ts` instead **spawns `claude -p`** to author autonomously and writes **stub
stories** (`# Stories: <idea>\n\n_Generated by engineer._`) that are **not** `Status: Accepted`.
That double-fails: it violates the non-negotiable *human-driven* rule, and the per-repo daemon
(which only builds `Status: Accepted` plans) would **skip the stubs anyway** — so the loop produces
nothing buildable. The spawned-subprocess path also introduced the session-id / `captureOnly` /
JSON-leak bugs patched repeatedly this program.

This redesign **keeps the 9.3 spine** (engineer = non-autonomous planner; spec→PR→merge=approval;
flywheel read; read-only governor; cross-repo isolation) and **reworks the execution model** to an
**agent-hosted, phone-drivable, human-gated DECIDE loop** that runs the *real* skills in the chat —
removing every spawned-`claude` routing/authoring/retro path. It also lands two contained
subsystems the shipped version lacked: a **pluggable intake port** (this phase wires the
`claude-session`/chat adapter only) and **daemon liveness** (pidfile-lock + 1-per-repo +
ensure-running), including the fix to the dead, mis-wired `launchDaemonDetached`.

**Scope decision (this PRD):** The `github-issues` adapter, the inbox poll/async buffer, and
**bidirectional write-back are deferred to a follow-on 9.3b** PRD. They are only meaningful together
(write-back has no target without an external source), and the core redesign is fully manual-
testable by typing an idea into the chat. The **intake port contract is defined here** so 9.3b is
purely additive — one adapter file + config, zero changes to routing/DECIDE/daemon.

## Goals & Non-Goals

**Goals**
- Replace the Node-TTY-REPL-spawning-`claude -p` engineer with an **agent-hosted interactive loop**:
  idea → route → **human-gated DECIDE in chat** → real `Status: Accepted` artifacts → spec PR →
  **ensure-running** the target daemon → loop. Phone-drivable.
- Author specs by running the **real DECIDE skills** (brainstorm→stories→plan) against the target
  repo on a `spec/<slug>` branch — the same artifacts a normal DECIDE phase produces.
- Define an **intake port + Envelope contract**; wire the **`claude-session` (chat) adapter** as the
  sole v1 source. Engineer core depends only on the port (loose coupling for 9.3b).
- **Daemon liveness:** `.daemon/daemon.pid` pidfile-lock created atomically (`O_EXCL`) as the
  **1-per-repo mutex**; liveness via `process.kill(pid,0)`; stale-pid reclaim.
- **ensure-running** (not manage): after landing artifacts for repo X, spawn a detached daemon iff
  none is alive; no-op if one is. Fix `launchDaemonDetached` (`--project` → `cwd: repoPath`) and
  wire it in (it is currently dead code).
- **Reuse the spine:** registry-driven routing (9.2), flywheel read of the 9.1 store, read-only
  governor reporting, cross-repo isolation.

**Non-Goals**
- **Deferred to 9.3b:** `github-issues` adapter, inbox poll / async capture buffer, bidirectional
  write-back (issue comment/transition). The port is designed to accept them additively.
- **No build, no auto-merge.** The human gate (spec PR merge) is non-negotiable; no code path runs a
  build without a merged spec PR.
- The engineer **does not manage/supervise/throttle/kill daemons** — `ensure-running` only.
- No governor *broker*/gating (per-daemon self-limit stands); no UI (9.4); no multi-host; no
  auto-applied harness self-edits (propose-only PRs through existing gates). **No Jira.**
- The 9.0 finish-time rebase is the **only** sanctioned rebase; this work adds none.

## Users / Personas

- **The human operator (James)** — drives the engineer from the phone: supplies ideas, confirms
  routing, runs the human-gated DECIDE, reviews/merges spec PRs.
- **The per-repo daemon** — downstream consumer of merged specs (9.0); unchanged except for the new
  pidfile lock it must create/respect.

## Functional Requirements

### A. Agent-hosted engineer loop (rework)

- **FR-1:** `conduct engineer` starts a **long-lived, agent-hosted interactive loop** (not a Node
  TTY REPL). On start it loads the registry (9.2 `RegistryReader`) and opens the 9.1 engineer store.
  *Negative:* if the registry or store is missing/unreadable, the loop reports it and continues in a
  degraded mode (empty known-projects / no lessons) rather than crashing.
- **FR-2:** Each iteration accepts a **feature idea** (free text); the loop persists context across
  ideas and continues until the operator exits cleanly.
- **FR-3:** For an idea, the engineer **proposes a target repo by reasoning in-chat against the
  registry** — **no spawned `claude -p`** — and the operator **confirms or redirects** before any
  authoring/write happens. *Negative:* if the operator redirects, no branch/PR is created in the
  originally-proposed repo.
- **FR-4 (edge):** One idea **may fan out to multiple repos**; the operator confirms the set, and
  authoring runs **per repo independently**.
- **FR-5 (edge):** If **no registered project fits**, the engineer offers to **create** one via 9.2
  `create` (scaffold + register) and routes the idea there. *Negative:* if the operator declines
  creation, the idea is dropped with no repo side effects.
- **FR-6 (the core fix):** Authoring runs the **real DECIDE skills** (brainstorm→stories→plan),
  **human-gated, in chat**, against the **target repo** on a `spec/<slug>` branch, producing real
  `Status: Accepted` artifacts. *Negative (regression guard):* the engineer **never** writes stub
  stories nor any artifact left at `Status: DRAFT`/un-accepted, and **never** spawns a subprocess to
  author.
- **FR-7:** The engineer **opens a spec PR** in the target repo; the operator's **merge is the
  approval**. The engineer **never builds** and **never auto-merges** (`gh pr merge` is never called).
- **FR-8:** **Flywheel read** — before/at planning, the engineer reads the 9.1 store and surfaces the
  **relevant** prior lessons (kickbacks/halts/retry-hotspots + narrative refs) for the target repo /
  similar features into the planning context.
- **FR-9:** **Read-only governor reporting** — the engineer reports aggregate token spend and
  kickback/halt/retry rates from the store; it does **not** gate or broker execution.
- **FR-10 (edge, non-negotiable):** There is **no path from idea → build without a merged spec PR**;
  the engineer never auto-merges; any harness self-edit it proposes is a PR through existing
  validation/no-auto-merge gates.
- **FR-11 (edge):** **Cross-repo isolation** — authoring for repo A creates the branch/PR **only** in
  A and never touches repo B's working tree, branches, or PRs.
- **FR-12 (measurability):** The flywheel is **measurable** — the engineer can compute whether
  kickback/halt/retry rates fall across successive engineer-planned features (per 9.1's metric).

### B. Intake port + claude-session adapter

- **FR-13:** Define an **intake port** whose only contract is the **Envelope**:
  `{ id, source, sourceRef, text, hintRepo?, status: pending|routed|deciding|done, receivedAt }`.
  The engineer core (routing/DECIDE) imports **only** the port — adding a source later is one adapter
  file + config with **zero** changes to routing/DECIDE/daemon.
- **FR-14:** A **`claude-session` adapter** (rides the chat; zero-friction capture) produces Envelopes
  from operator chat input: `source = "claude-session"`, `sourceRef` identifies the originating chat
  turn. This is the **sole** adapter wired in this phase.
- **FR-15 (edge, idempotency):** `source` + `sourceRef` provide idempotency — re-submitting an idea
  with the same `sourceRef` must **not** create a second in-flight Envelope / double-process.
- **FR-16 (negative):** An Envelope missing a required field (e.g. empty `text`) is **rejected with a
  clear error**, not silently dropped or processed as blank.

> *Design boundary:* on-disk inbox buffering and the async **capture/processing split** exist to
> serve async sources (github poll) and are realized in **9.3b**. The `claude-session` path processes
> synchronously through the same port; the seam (port + Envelope) is what this phase locks.

### C. Daemon liveness, 1-per-repo & ensure-running

- **FR-17:** **Liveness primitive = pidfile lock.** `.daemon/daemon.pid` holds
  `{ pid, uuid, startedAt }`, created **atomically with `O_EXCL`**; the atomic create **is** the
  1-per-repo mutex.
- **FR-18:** **Liveness check** = `process.kill(pid, 0)` (true alive/dead, no heartbeat threshold).
- **FR-19 (edge):** A **stale pidfile** (file exists but pid is dead — e.g. after `kill -9`, possibly
  leaving a half-built worktree) is detected, the lock **reclaimed**, and a fresh daemon spawned.
  The repo is **never** permanently refused because of a stale lock.
- **FR-20 (edge, the race):** Two concurrent daemon starts for one repo → **exactly one** wins the
  `O_EXCL` create; the other **no-ops**. Two daemons never run in one repo.
- **FR-21:** **`ensure-running` (not manage):** after landing artifacts for repo X, the engineer
  checks liveness for X — **none/stale** → spawn a **detached, fire-and-forget** daemon;
  **alive** → **no-op** (the running daemon picks up new work on its next poll). The engineer never
  throttles/kills/supervises; in practice it spawns **once per repo lifetime**.
- **FR-22 (the launch fix):** `launchDaemonDetached()` currently spawns
  `npx conduct daemon --project <path>`, but **`--project` does not exist** (the daemon reads its
  working directory). Fix it to spawn the daemon with **`cwd: repoPath`** using the real CLI form,
  and **wire it into `ensure-running`** (it is currently dead, never-called code).
- **FR-23 (non-authoritative):** The engineer **may** mirror daemon state into the registry
  `daemonState` field **purely for the governor's reporting**; the **pidfile is the source of truth**
  and registry state is never trusted for the liveness/mutex decision.

### D. Handoff invariant (asserted, reused)

- **FR-24:** The daemon builds only specs present on `main` (it scans the working tree); therefore
  **merging the spec PR is the handoff signal** (human gate #2). The build-ready predicate is
  unchanged: plan exists + stories `Status: Accepted` (no DRAFT) + plan has a dependency tree + slug
  not already in `.daemon/processed/`.

## Non-Functional Requirements

- **Non-autonomy by construction:** every path to "a daemon builds" passes through a human-merged
  spec PR. No code path from engineer → build skips that merge.
- **Reuse over reinvention:** routing reuses the 9.2 registry reader; lessons reuse the 9.1 store;
  authoring reuses the DECIDE skills and the existing PR machinery; no parallel planning stack.
- **Loose coupling at the intake seam:** engineer core depends only on the port interface; adapters
  are swappable/additive.
- **Cross-repo safety:** the engineer resolves the target repo from the registry's canonical path and
  confines all writes (branch/PR/pidfile) to it.
- **Crash safety:** liveness/lock logic is correct under `kill -9` and concurrent starts (no
  deadlock, no double-daemon, no permanent refusal).
- **Read-only where it consumes:** read-only over the store/registry except (a) creating a project via
  9.2 `create`, and (b) the optional non-authoritative `daemonState` mirror.

## Acceptance Criteria / Success Metrics

Every FR covered by tests (vitest + real git repos for cross-repo authoring/PR and liveness paths):

1. **Loop:** `conduct engineer` starts agent-hosted, loads registry + store, accepts an idea, loops,
   exits cleanly; missing registry/store → degraded, no crash (FR-1/2).
2. **Routing:** proposes a repo from registry+idea **without spawning `claude -p`**, requires
   confirmation; redirect leaves the original repo untouched; multi-repo fan-out authors per repo;
   no-fit → offers `create`, decline → no side effects (FR-3/4/5/11).
3. **Authoring (the fix):** brainstorm→stories→plan run in the **target** repo on `spec/<slug>`,
   producing real `Status: Accepted` artifacts; **a stub/DRAFT story or a spawned authoring
   subprocess is a test failure**; a spec PR is opened; **no build runs; no `gh pr merge`** (FR-6/7/10).
4. **Flywheel:** seeded store signals surface as relevant lessons in planning context; rates are
   computable across successive features (FR-8/12).
5. **Governor:** computes aggregate spend + kickback/halt/retry rates read-only (FR-9).
6. **Intake port:** Envelope contract enforced; `claude-session` adapter yields a valid Envelope;
   same `sourceRef` does not double-process; empty `text` rejected (FR-13/14/15/16).
7. **Liveness:** `O_EXCL` create is the mutex; concurrent starts → exactly one daemon (FR-17/18/20);
   `kill -9` stale pidfile → reclaim + respawn, never permanent refusal (FR-19); `--project` replaced
   by `cwd: repoPath` and `ensure-running` actually calls it; alive daemon → no-op (FR-21/22).
8. **Handoff:** unmerged spec PR → daemon does not build; merged spec on main → build-ready predicate
   passes (FR-24).
9. Existing tests pass; harness integrity green; CHANGELOG `[Unreleased]` + README/conductor README +
   HARNESS.md updated; the superseded patches are not reintroduced.

## Scope

### In Scope
- Agent-hosted engineer loop (idea intake, in-chat routing, human-gated DECIDE, spec branch+PR, exit).
- Removal of the spawned-`claude -p` routing/authoring/retro-narrative paths and stub-story writing.
- Intake **port + Envelope** contract; **`claude-session`** adapter only.
- Daemon **pidfile-lock liveness + 1-per-repo + stale reclaim**; **ensure-running**; the
  `launchDaemonDetached` `--project`→`cwd` fix wired in.
- Flywheel read (9.1) + read-only governor (reused). Tests + docs.

### Out of Scope (→ 9.3b unless noted)
- `github-issues` adapter; inbox poll / async capture buffer; **bidirectional write-back**.
- Spawning/managing daemons beyond fire-and-forget ensure-running; build; auto-merge; governor
  broker/gating; UI (9.4); multi-host; auto-applied harness self-edits; Jira.

## Key Decisions & Rationale

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | **Agent-hosted loop** replaces the Node-REPL-spawning-`claude -p` | Restores human-driven DECIDE (FR-6 intent); removes session-id/`captureOnly`/JSON-leak bug class at the root. |
| 2 | **Run the real DECIDE skills**, produce `Status: Accepted` artifacts | Stubs were un-buildable and violated the gate; consistency + dogfooding; one planning path. |
| 3 | **Split:** keep `claude-session`; defer github-issues + write-back to 9.3b | Write-back is meaningless without an external source; the chat path is fully manual-testable now; the **port** keeps 9.3b additive. |
| 4 | **Pidfile lock (`O_EXCL`)** over registry-heartbeat for liveness/mutex | Atomic create *is* the mutex; `kill(pid,0)` is exact (no heartbeat-threshold guessing); registry stays non-authoritative. |
| 5 | **ensure-running, not manage** | Daemons stay independent/fault-isolated; engineer holds no lifecycle ownership; one spawn per repo lifetime. |
| 6 | **Fix `launchDaemonDetached` to `cwd: repoPath`** and wire it in | `--project` flag doesn't exist; the function was dead code; this is the actual launch bug. |
| 7 | **Flywheel read + read-only governor reused as-is** | The whole point of 9.1; a broker is unjustified at solo scale. |

  approval). The `O_EXCL`-single-winner mutex is the v1 design; a later phase may revisit the
  race/coordination model (e.g. multi-worker daemons or a different locking primitive). Treat FR-20
  as the current-iteration contract, not a permanent one — keep the liveness/lock logic isolated so
  it can be swapped without disturbing the loop or intake port.
