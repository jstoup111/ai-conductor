## Summary

Builds the non-autonomous `conduct engineer` interactive loop: idea â†’ route (registry, human-confirmed) â†’ author spec via DECIDE in the target repo on a `spec/<feature>` branch â†’ open spec PR; with a flywheel read (9.1 store), a read-only governor, and a measurable learning trend. **34 tasks.** The engineer never builds and never auto-merges; cross-repo safety and non-autonomy are structural, not vigilance-based.

## Technical Approach

Layered, bottom-up so each layer is independently testable before the loop wires them together:

1. **Runtime readers** (new, types exist only): `RegistryReader` impl over `readRegistry`/`canonicalizePath` (`registry.ts`); `EngineerStoreReader` impl reading `signals.jsonl` with the `report-renderer.ts` resilient-parse convention (skip malformed lines).
2. **Domain types** (ADR-007/004): routing outcome **discriminated union** (`confirmed`/`redirected`/`create`/`declined`, no booleans, exhaustive switch); `TargetRepo` parsed once from the registry canonical path and threaded thereafter.
3. **Shared rate fn** (FR-9 + FR-12, ADR-006): one `computeSignalRates(signals)` over `EngineerSignal[]` built on the existing `aggregate*` helpers â€” kickback/halt/retry rates per 9.1's metric + token aggregate; `<2` â†’ insufficient; zero-denominator excluded (no div-by-zero). **Both** governor and trend call it â€” no second implementation.
4. **LessonStore port** (ADR-006): `record`/`retrieve` with `LessonQuery {text, namespace, topK?, filters?, crossProject?}` and `RetrievedLesson {id, text, score?, metadata, validAt?}`; default adapter = keyword/recency over JSONL (JSONL stays system of record); `selectLessons(idea, project, store) â†’ LessonDigest`. Composite `project:feature` namespace string + filters; `score?` optional (rank-by-position fallback).
5. **Authored-keys ledger** (ADR-006 provenance): durable `(project, feature)` set written at spec-PR open; FR-12 trend over `store âˆ© ledger`. No 9.1 schema change.
6. **Routing** (ADR-007): LLM inference over registry Ã— idea (via `provider.invoke`) â†’ ranked candidates + rationale; affirmative-confirm gate; create-on-no-fit via 9.2 `create`.
7. **Authoring** (ADR-004): resolve target canonical path (stale â†’ error, **no cwd fallback**); subprocess DECIDE with `cwd = target` via `claude-provider.invoke({cwd})`; branch `spec/<slug>` off default branch (`originDefaultBranch`, never hardcode `main`); dirty-tree guard; existing-branch suffix; spec artifacts only (no impl).
8. **PR handoff** (ADR-007): reuse PR machinery against the target cwd; report URL; no-remote â†’ non-fatal PR-skip; **never** `gh pr merge`, **never** build.
9. **Daemon launch detached** (ADR-005): fire-and-forget `spawn(detached:true, stdio:'ignore')`; no retained handle/IPC/supervision state.
10. **Non-autonomy structural guarantee** (ADR-005/FR-10): test asserts the engineer module imports neither build/pipeline nor merge entry points, and any launch is detached.
11. **CLI loop** (FR-1/FR-2): `engineer` subcommand in `cli.ts` + detection in `index.ts` (mirroring `detectRegistryCommand`); load registry+store, project count; idea intake loop with context persistence, blank-idea re-prompt, per-idea error isolation, clean EOF/exit.

**Sequencing rationale:** readers + types + rate fn + port are leaf dependencies of everything else (build first); routing/authoring/PR are the idea-processing spine; the CLI loop is wired last so it composes finished, tested units. Structural non-autonomy test (Task 30) and cross-repo isolation tests (Task 21) are the two High-impact-risk gates from the architecture review.
