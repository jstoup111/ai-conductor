# Implementation Plan: Phase 9.0 — Rebase-on-Latest + Conflict→HALT

**Date:** 2026-06-25
**Design:** `.docs/specs/2026-06-25-phase-9.0-rebase-on-latest.md`
**Stories:** `.docs/stories/phase-9.0-rebase-on-latest.md`
**ADR:** `adr-001-rebase-insertion-mechanism.md` (APPROVED — Option A)
**Conflict check:** Clean as of 2026-06-25 (FR-5×FR-7 resolved)

## Summary

Adds a `rebase` gate-loop step (engine-native, like `complexity`) that, before `finish`, rebases
the daemon worktree branch onto the discovered origin default branch, re-verifying via the
existing kickback machinery on code/test changes, auto-resolving only CHANGELOG conflicts, and
HALTing (worktree kept, rebase paused) on all others. ~16 tasks, all in `src/conductor` + docs.

## Technical Approach

Per ADR-001 (Option A), `rebase` is a new `loopGate: true` step in `ALL_STEPS` with **no
`skillName`** — the engine handles it natively (mirroring how `complexity` is engine-handled and
explicitly not dispatched, `step-runners.ts:250`). `finish`'s prerequisite is repointed from
`retro` to `rebase` so finish cannot run until the branch is current.

The step's **gate verdict is the design's keystone**: *satisfied ⇔ the branch is already current
with the discovered base*. This makes FR-4 (no-op) the satisfied state, so after a kickback
re-runs `build`, the selector re-enters `rebase`, finds the branch current, marks it satisfied,
and proceeds to `finish` — **no re-invalidation, no `MAX_GATE_SELECTIONS` false HALT**
(architecture-review condition #2). A genuinely stale branch must never report satisfied.

New code lives in a focused `engine/rebase.ts` module (pure helpers: base discovery+fetch with
local fallback, rebase execution, diff classification, CHANGELOG resolution, HALT writing,
verdict computation) consumed by a native dispatch branch in `conductor.ts`. Invalidation reuses
`gate-verdicts.ts` (write `satisfied:false` kickback verdicts for `build`/`manual_test`) and the
existing `selector.ts` routing. Events extend the existing `ConductorEvent` union. All git work
uses the `execa` pattern already in `daemon-deps.ts`. Tests are vitest, matching
`test/integration/gate-loop.test.ts` (mock `execa`/runner, seed state, assert verdicts/events).
