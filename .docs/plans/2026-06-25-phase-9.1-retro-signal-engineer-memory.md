# Implementation Plan: Phase 9.1 — Retro Signal + Engineer Memory Store

**Date:** 2026-06-25
**Design:** `.docs/specs/2026-06-25-phase-9.1-retro-signal-engineer-memory.md`

**Conflict check:** Clean as of 2026-06-25 (FR-7×ST-024, FR-5/6×ST-005 resolved)

## Summary

Adds a `engineer-store` module + an emission step wired into the daemon's feature completion: on
daemon `done`/`halted`, assemble a structured signal (reusing report-renderer), produce a
narrative (Option A: emission owns it), and atomically append to `~/.ai-conductor/engineer/`. Best-
effort and concurrency-safe. ~14 tasks, all in `src/conductor` + docs.

## Technical Approach

New `src/engine/engineer-store.ts`: `resolveEngineerDir()` (path + `$AI_CONDUCTOR_ENGINEER_DIR` override +
auto-create), a `EngineerSignal` type, `assembleSignal()` (parse the feature's `events.jsonl` + reuse
`report-renderer` aggregation + `FeatureOutcome`), `appendSignal()` (one atomic `O_APPEND` write),
and `writeNarrative()` (`narratives/<project>/<feature>-<runId>.md`). A `produceNarrative()` helper
generates the `done` narrative via the existing LLM provider (the work the in-loop retro did) and a
template-assembled short halt narrative for `halted` (no Claude call needed). `emitEngineerSignal()`
wraps it all best-effort (log + swallow). Wired into `daemon-runner.ts makeRunFeature` **after
`readOutcome`, before `teardownWorktree`**, gated on daemon mode. The gate loop skips the in-loop
`retro` step under the daemon (manual runs unchanged). A types-only stub reader interface is
exported for 9.3.
