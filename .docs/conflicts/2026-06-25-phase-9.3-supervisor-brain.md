## Degrading overlap (RESOLVED): FR-9 × FR-12 — two rate computations

**Stories involved:** "Read-only governor reporting" (FR-9) vs "Flywheel is measurable across
successive brain-planned features" (FR-12)
**Files:** `.docs/stories/phase-9.3-supervisor-brain.md` (both)
**Type:** overlap (resource/logic contention)
**Severity:** degrading

**Description:** Both stories compute kickback/halt/retry rates from the 9.1 store — FR-9 as an
aggregate snapshot, FR-12 as a per-feature ordered trend. Two independent implementations would
drift and could report inconsistent numbers for the same signals.

**Resolution options:**
1. Both consume **one shared rate-computation** that reuses 9.1 FR-9's metric definition. *(least
   disruptive)*
2. Let each compute independently and reconcile in tests.
3. Push the computation down into a 9.1 store-reader helper.

**Resolution applied:** **Option 1** (with a nod to 3 at architecture). FR-9 gains a happy path +
Done-When that the rate computation is a single shared function aligned to 9.1's metric; FR-12's
Done-When references that shared function. Stories updated in place.

## Degrading gap (RESOLVED): FR-12 × 9.1 signal schema — brain-planned provenance

**Stories involved:** "Flywheel is measurable… brain-planned features" (FR-12) vs Phase 9.1 "Write
the structured signal record" (9.1 FR-3) / "Signals support cross-feature rate metrics" (9.1 FR-9)
**Files:** `phase-9.3-supervisor-brain.md` vs `phase-9.1-retro-signal-brain-memory.md`
**Type:** state-conflict (assumed data not produced upstream)
**Severity:** degrading

**Description:** FR-12 must measure the trend over **brain-planned features specifically** and
**exclude** non-brain daemon work. But the 9.1 signal schema
(`{schemaVersion, ts, project, feature, runId, outcome, kickbacks[], halts[], retryHotspots[],
tokens, durationByStep, narrativeRef}`) carries **no field marking a signal as brain-planned** — the
daemon emits a signal for *every* feature it builds, however it was planned. Without disambiguation
the trend would be contaminated by manually-planned work. (Same class as the resolved 9.1 FR-5/6 ×
ST-005 retro-skip gap and the 9.2 `create`-vs-`bootstrap` provenance gap.)

**Resolution options:**
1. Brain keeps its **own record of authored `(project, feature)` keys** (the specs it opened,
   FR-6/FR-7) and **intersects** with store signals by `(project, feature)` — **no 9.1 change**.
   *(least disruptive; self-contained in 9.3)*
2. Add a `source`/`planner` provenance field to the 9.1 signal schema (cross-phase change).
3. Infer provenance from `spec/<feature>` branch naming conventions. *(brittle)*

**Resolution applied:** **Option 1.** FR-12 gains a negative path + Done-When stating the
brain-planned set is derived from the brain's authored keys and intersected with store signals,
requiring no 9.1 schema change; a feature with no emitted signal yet is simply absent (no fabricated
zero). **Architecture/ADR flag:** whether to *additionally* add a provenance marker to the 9.1
signal (Option 2) is deferred to architecture-review; until then the intersection is the source of
truth.
