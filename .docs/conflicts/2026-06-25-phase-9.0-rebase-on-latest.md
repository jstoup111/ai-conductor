## Degrading overlap (RESOLVED): FR-5 × FR-7 — docs-only change triggers re-verify

**Stories involved:** "Invalidate downstream gates on a file-changing clean rebase" (FR-5) vs
"Auto-resolve a CHANGELOG.md conflict safely" (FR-7)
**Type:** overlap
**Severity:** degrading

**Description:** FR-7 auto-resolving a CHANGELOG conflict changes a file. A naive FR-5 ("any
changed file → invalidate build") would then force a full build/manual_test re-verify for a
**docs-only** change — wasteful, not incorrect.

**Resolution options:**
1. Scope FR-5's trigger to **code/test path changes**; docs-only (CHANGELOG) changes do not
   invalidate. *(least disruptive)*
2. Accept the redundant re-verify on docs-only changes.
3. Diff changed paths against a code/test allowlist before invalidating.

**Resolution applied:** **Option 1** (functionally the same as 3). FR-5's happy path and a new
negative path now state the invalidation trigger is a change to **code/test paths**, and FR-7
gains a Done-When that a CHANGELOG-only auto-resolution does not invalidate build/manual_test.
Stories updated in place.

## Benign safety overlap (NO ACTION): FR-6 × ST-023 fresh verification

`finish`'s fresh verification (`ST-023`) re-runs the full suite, overlapping FR-6's post-rebase
re-verify. This is **defense-in-depth**, not a conflict: FR-6 routes post-rebase failures back
through the autoheal/anti-oscillation loop *before* `finish` (so simple breaks self-heal),
while `finish`'s run is the final hard gate. They are complementary. No change needed.

---

## Design constraint recorded (NOT a conflict): rebase idempotency vs anti-oscillation

The rebase step invalidates `build` when it changes code/test files; the loop then re-runs build
and returns to the rebase point. To avoid repeatedly invalidating and tripping the existing
`MAX_GATE_SELECTIONS` anti-oscillation HALT, **the rebase step must be a no-op once the branch is
current** (already on the new base). This is covered by FR-4 (no-op → skip re-verify). Flagged
for architecture-review/plan to ensure the implementation re-checks "already current" each pass
rather than unconditionally re-attempting the rebase. No story change required.

- **Paused-rebase worktree vs idempotent `createWorktree` reuse** — `ST-007` already specifies
  "detect the existing worktree and reuse it." FR-9's resume path is consistent. No conflict.
- **CHANGELOG auto-resolution vs finish/PR CHANGELOG handling** — no finish/PR code edits

- **Finish "git clean" gate (`ST-006`/`ST-023`) vs rebase** — a successful rebase (incl. a
  `--continue` after CHANGELOG resolution) leaves a clean, committed tree, so the finish gate
  still holds. The HALT path never reaches finish. No conflict.
