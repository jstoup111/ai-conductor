# Retro: Phase 9.2 — Project Registry + Creation

**Date:** 2026-06-25 | **Stats:** 13-task plan, 1 rework cycle (FR-11 credential leak) + 1 git-recovery incident, 1088 tests passing

## Part A: Harness

- **H-1 (RECURRING — now 3/3 features; the dominant trend):** A correctness/security derivation
  was correct in isolation but a **sibling call site bypassed it**, untested with real adversarial
  input — passing a green suite, caught only by the evaluator. Here: `redactRemote` was correct and
  unit-tested, and `register` used it, but `create --remote` wrote the **raw token URL** to
  `~/.ai-conductor/registry.json` (a real credential to disk). The FR-11 story existed and the
  redaction helper had tests — but **no test fed `create` a token URL**. Same shape as 9.0
  (predicate tested on a clean tree, not the in-progress state) and 9.1 (project key injected, not
  derived). **Severity:** High (security). **Fix (sharper still):** the RED-spec step must
  enumerate **every call site** of a security/correctness derivation and test each against real
  adversarial inputs — not just the helper + one call site. Add to `writing-system-tests`: "for a
  redaction/validation/derivation helper, assert it at EVERY call site, with a real malicious input
  (token URL, traversal path, etc.), not only in the helper's own unit test."
- **H-2 (tooling/process — new):** Feature branches were **auto-rebased onto `origin/main`
  mid-work** when sibling PRs merged (hit both the 9.1 and 9.2 branches), leaving the 9.2 branch in
  a mid-rebase CHANGELOG conflict that blocked a commit. Not a git hook/config (verified: no
  `pull.rebase`/`rebase.autostash`, default `core.hooksPath`, only `.sample` hooks). Likely a
  subagent's "keep-branch-current" rebase or external automation. **Severity:** Medium.
  **Fix:** (a) implementation-agent prompts must say **do NOT fetch/rebase/pull — stay on the
  branch as-is**; (b) investigate the automation source. **Not in conflict with 9.0:** the 9.0
  rebase-on-latest is the *only* sanctioned rebase — daemon-driven, **at `finish` only**, with
  conflict→HALT and the **FR-7 CHANGELOG auto-resolver**. The incident was an *uncontrolled* rebase
  *outside* that machinery (the daemon isn't even running in manual SDLC dev). Tellingly, the
  conflict was a CHANGELOG `[Unreleased]` collision — exactly what FR-7 auto-resolves; had it gone
  through the 9.0 path it would have resolved automatically. 9.0 is the fix for this pain, not the
  cause; the rule is simply "no rebase except the controlled 9.0 one."
- **H-3 (my slip):** A commit landed with the **wrong message** because I reused a stale
  `-F <msgfile>` path (the old hook-fix message). Caught + amended. **Severity:** Low.
  **Fix:** use a per-commit message file (or inline), never a reused generic path.

**Proposed changes:**
- [ ] H-1: `skills/writing-system-tests/SKILL.md` — "assert a derivation at every call site with
      real adversarial inputs"; conduct gate checks redaction/validation helpers have a call-site test.
- [ ] H-2: add "do not fetch/rebase/pull" to implementation-agent dispatch prompts; investigate the
      mid-build rebase source.
- [ ] H-3: per-commit message files only.

## Part B: Application

- **A-1:** `create` leaves an **orphan scaffold** (git repo + files) if the registry write fails
  (unwritable dir) — no record, no rollback. Not corruption (re-run dedups), evaluator-rated minor.
  **Fix (story):** on registry-write failure after a successful scaffold, either roll back the
  freshly-created dir or print a "scaffolded but not registered; run `conduct register`" hint.
- **A-2:** `create` with **no name** falls through to the interactive pipeline (cosmetic). **Fix:**
  guard `detectRegistryCommand` to error on `create` with no positional name.

**Proposed changes:**
- [ ] A-1: story — handle scaffold-succeeded-but-register-failed (rollback or actionable hint).
- [ ] A-2: story — `create` with no name → clear usage error, not interactive fallthrough.

## Part C: Context Efficiency

- **C-1:** This cycle I added explicit **real-input** RED-spec instructions (vs 9.1) — it improved
  coverage but the leak still slipped because the instruction said "test the derivation with real
  inputs" without "**at every call site**." The fix is a one-line sharpening (H-1), not more tokens.
- **C-2:** The git-recovery incident (H-2) cost real wall-clock + tokens mid-build. Preventing the
  mid-build rebase (H-2) removes that cost entirely.

**Proposed changes:**
- [ ] C-1: the H-1 SKILL.md sharpening (call-site coverage) is the high-leverage, low-token fix.
