# Retro: Phase 9.1 — Structured Retro Signal + Brain Memory Store

**Date:** 2026-06-25 | **Stats:** 14-task plan, 1 rework cycle (FR-9 critical fix), interventions = necessary only (scope, narrative-storage steer, ADR approval), 1052 tests passing

## Part A: Harness

- **H-1 (RECURRING — now a trend):** A story's **negative path** was implemented but its test
  exercised an **injected/stubbed stand-in** instead of the **real wiring**, so a CRITICAL bug
  passed a green suite — caught only by the evaluator. Here: FR-9 cross-project disambiguation.
  `deriveProjectName(worktree)` returned `.worktrees` for every real daemon run, but the unit test
  passed `project` explicitly and the integration test seeded worktrees under a `mkdtemp` base —
  neither exercised the real derivation. **This is the same shape as 9.0's critical** (the
  satisfied-predicate was tested on a clean tree, not the in-progress state). **Severity:** High.
  **Fix (sharper than 9.0's H-1):** for a correctness-critical **derivation/predicate**, a negative
  test must drive the **real production path** with **realistic inputs** (a real worktree layout,
  the real project root) — not pass the value-under-test as an injected input or stub. Add to
  `writing-system-tests`/RED-spec guidance: "for each derived value, assert it from the real
  derivation, never from an injected literal."
- **H-2 (process):** BUILD was again a **single large delegated implementation agent**, not the
  per-task pipeline loop. 9.0's retro C-1 proposed using the per-task loop for 9.1 precisely so a
  domain reviewer would see each correctness predicate (e.g. the project derivation) during
  implementation. I didn't switch. The evaluator backstop worked, but the earlier gate is cheaper.
  **Severity:** Medium. **Fix:** for 9.2+, actually route BUILD through per-task dispatch with a
  domain-review step on each derivation/predicate task.

**Proposed changes:**
- [ ] H-1: `skills/writing-system-tests/SKILL.md` + `skills/conduct` gate — require derived/critical
      values to be asserted via the real production path, not injected literals.
- [ ] H-2: adopt per-task BUILD dispatch (with per-task domain review) starting 9.2.

## Part B: Application

- **A-1:** `project` key now flows correctly, but `FeatureRunnerDeps` carries a **basename string**,
  not a structured project identity. If 9.2's registry introduces a canonical project id, this
  string key becomes a second source of truth. **Severity:** Low. **Fix (story for 9.2):** when the
  registry lands, derive the brain-store `project` key from the registry's canonical id.
- **A-2:** `produceNarrative` `done` path makes a real `provider.invoke` at emission time (outside
  the loop). No timeout/size cap on that call is enforced here beyond the provider's own. **Severity:**
  Low. **Fix:** confirm the provider's timeout applies to emission-time calls; add a cap if not.

**Proposed changes:**
- [ ] A-1: tracked for 9.2 (registry-canonical project id → brain key).
- [ ] A-2: verify provider timeout covers emission-time narrative calls.
